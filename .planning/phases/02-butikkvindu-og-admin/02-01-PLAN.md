**IMPORTANT:** The root layout already renders Header, Footer, SkipLink, and `<main id="main-content">`. The (public) route group layout must NOT duplicate these. It should be a minimal pass-through layout or just provide the route group boundary.

**A. Create src/app/(public)/layout.tsx** — A simple pass-through layout:
```tsx
export default function PublicLayout({ children }: { children: React.ReactNode }) {
  return <>{children}</>
}
```

However, there is a problem: the root layout wraps children in `<main id="main-content" className="mx-auto max-w-[1200px] px-4 lg:px-8">`. This works for most public pages but experience detail pages need full-width hero images. To handle this:
- Keep root layout as-is for now (it provides the standard container).
- For experience and article detail pages that need full-width heroes, the hero component itself must break out of the container using negative margins or the page must opt out. The simplest approach: the HeroImage component uses `className="-mx-4 lg:-mx-8"` with a `max-w-none` wrapper to break out of the parent container, then the content below returns to the normal flow.

Actually, the better architecture: change the root layout to NOT wrap children in a constrained `<main>`. Instead, each page or route group layout should control its own max-width. This gives full-width hero images AND contained content. The `<main id="main-content">` tag should still exist but without the max-width constraint. Each page component adds its own container.

**ROOT LAYOUT MODIFICATION (src/app/layout.tsx):**
Change the `<main>` from:
```tsx
<main id="main-content" className="mx-auto max-w-[1200px] px-4 lg:px-8">
```
to:
```tsx
<main id="main-content">
```

Then each public page controls its own container width internally. This is necessary because:
- /opplevelser/[slug] needs a full-width hero image
- /blogg/[slug] needs a full-width hero image
- /produkter needs a max-w-1200px container
- Admin pages need a completely different layout

**1. Create src/app/sitemap.ts** (CONT-05) — Dynamic sitemap using Next.js App Router convention:
```typescript
import type { MetadataRoute } from 'next'
import { getProducts } from '@/lib/data/products'
import { getExperiences } from '@/lib/data/experiences'
import { getArticles } from '@/lib/data/articles'

const BASE_URL = process.env.NEXT_PUBLIC_SITE_URL || 'https://rootsculture.no'

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const [products, experiences, articles] = await Promise.all([
    getProducts(),
    getExperiences(),
    getArticles(),
  ])

  const staticPages: MetadataRoute.Sitemap = [
    { url: BASE_URL, lastModified: new Date(), changeFrequency: 'weekly', priority: 1 },
    { url: `${BASE_URL}/produkter`, lastModified: new Date(), changeFrequency: 'daily', priority: 0.9 },
    { url: `${BASE_URL}/opplevelser`, lastModified: new Date(), changeFrequency: 'daily', priority: 0.9 },
    { url: `${BASE_URL}/blogg`, lastModified: new Date(), changeFrequency: 'daily', priority: 0.8 },
    { url: `${BASE_URL}/om-oss`, changeFrequency: 'monthly', priority: 0.5 },
    { url: `${BASE_URL}/kontakt`, changeFrequency: 'monthly', priority: 0.5 },
  ]

  const productPages: MetadataRoute.Sitemap = products.map((p) => ({
    url: `${BASE_URL}/produkter/${p.slug}`,
    lastModified: p.updatedAt,
    changeFrequency: 'weekly' as const,
    priority: 0.7,
  }))

  const experiencePages: MetadataRoute.Sitemap = experiences.map((e) => ({
    url: `${BASE_URL}/opplevelser/${e.slug}`,
    lastModified: e.updatedAt,
    changeFrequency: 'weekly' as const,
    priority: 0.7,
  }))

  const articlePages: MetadataRoute.Sitemap = articles.map((a) => ({
    url: `${BASE_URL}/blogg/${a.slug}`,
    lastModified: a.updatedAt || a.publishedAt || undefined,
    changeFrequency: 'monthly' as const,
    priority: 0.6,
  }))

  return [...staticPages, ...productPages, ...experiencePages, ...articlePages]
}
```
