**1. Create src/lib/firebase/storage.ts** — Client-side Firebase Storage upload helper (ADMN-07):
```typescript
'use client'
import { ref, uploadBytesResumable, getDownloadURL } from 'firebase/storage'
import { storage } from '@/lib/firebase/client'

export interface UploadProgress {
  percent: number
  status: 'uploading' | 'complete' | 'error'
  url?: string
  error?: string
}

export async function uploadImage(
  file: File,
  path: string,
  onProgress?: (progress: UploadProgress) => void
): Promise<string> {
  // Validate file type and size client-side
  const allowedTypes = ['image/jpeg', 'image/png', 'image/webp']
  if (!allowedTypes.includes(file.type)) {
    throw new Error('Ugyldig filformat. Bruk JPG, PNG eller WebP.')
  }
  if (file.size > 5 * 1024 * 1024) {
    throw new Error('Filen er for stor. Maks 5 MB per bilde.')
  }

  const storageRef = ref(storage, path)
  const uploadTask = uploadBytesResumable(storageRef, file)

  return new Promise((resolve, reject) => {
    uploadTask.on('state_changed',
      (snapshot) => {
        const percent = Math.round((snapshot.bytesTransferred / snapshot.totalBytes) * 100)
        onProgress?.({ percent, status: 'uploading' })
      },
      (error) => {
        onProgress?.({ percent: 0, status: 'error', error: error.message })
        reject(error)
      },
      async () => {
        const url = await getDownloadURL(uploadTask.snapshot.ref)
        onProgress?.({ percent: 100, status: 'complete', url })
        resolve(url)
      }
    )
  })
}

export function deleteImage(path: string): Promise<void> {
  // Import deleteObject here to avoid pulling it in unnecessarily
  return import('firebase/storage').then(({ deleteObject, ref: storageRef }) => {
    return deleteObject(storageRef(storage, path))
  })
}
```

**3. Create admin layout and components** — All components follow 02-UI-SPEC exactly:

**src/app/admin/layout.tsx** — Server Component wrapper. The middleware already protects /admin routes (ADMN-01), so this layout does not need redundant auth checks. However, for defense-in-depth, call `verifySession()` from DAL and check role === 'admin'. If not admin, redirect to '/'. This layout wraps children in AdminShell. It does NOT render the root layout's Header/Footer — admin has its own chrome. BUT: the root layout always renders Header+Footer. To avoid showing Header/Footer on admin pages, admin layout must visually hide them or we need a different approach.

**IMPORTANT ARCHITECTURE DECISION:** The root layout (`src/app/layout.tsx`) currently renders Header and Footer for ALL routes. Admin pages should NOT show the public Header/Footer. The cleanest approach: use a route group. Move the Header and Footer rendering into a `(public)` layout instead of root layout. However, this would require restructuring the root layout.

The pragmatic approach for Phase 2: The admin layout renders AdminShell which takes over the full viewport with `position: fixed; inset: 0; z-index: 200` to overlay the root Header/Footer. This works because:
- Admin is always full-screen
- No SEO concern for admin pages
- Root layout's Header/Footer are simply hidden behind the admin overlay

```tsx
// src/app/admin/layout.tsx
import { redirect } from 'next/navigation'
import { verifySession } from '@/lib/dal'
import { AdminShell } from '@/components/admin/AdminShell'

export const metadata = { title: 'Admin — Roots & Culture' }

export default async function AdminLayout({ children }: { children: React.ReactNode }) {
  const session = await verifySession()
  if (!session || session.role !== 'admin') {
    redirect('/')
  }
  return <AdminShell>{children}</AdminShell>
}
```

**A. Create Server Actions** — All actions use `'use server'`, verify admin session, validate with Zod, write to Firestore via adminDb, and call `revalidateTag` to bust cache.

**src/actions/products.ts** (ADMN-02):
```typescript
'use server'
import { revalidateTag } from 'next/cache'
import { verifySession } from '@/lib/dal'
import { adminDb } from '@/lib/firebase/admin'
import { productSchema } from '@/lib/validations'
import { FieldValue } from 'firebase-admin/firestore'
```

Functions:
- `createProduct(formData: FormData)` — Parse FormData, validate with productSchema, verify admin session. Write to `products` collection with `adminDb.collection('products').add({...})`. Set `publishedAt` to `FieldValue.serverTimestamp()` if `publish` is true, else `null`. Set `inStock` to `stockCount > 0`. Set `createdAt` and `updatedAt` to `FieldValue.serverTimestamp()`. Call `revalidateTag('products')`. Return `{ success: true, id }` or `{ success: false, errors }`.
- `updateProduct(id: string, formData: FormData)` — Same validation flow. Use `adminDb.collection('products').doc(id).update({...})`. Set `updatedAt`. Call `revalidateTag('products')` and `revalidateTag(slug)`.
- `deleteProduct(id: string)` — Verify admin. Delete doc. Call `revalidateTag('products')`. Return `{ success: true }` or error.
- `getAllProducts()` — Server-side function for admin list (no publishedAt filter — admin sees drafts too). Reads from Firestore directly (no cache). Returns `Product[]`.

IMPORTANT: These Server Actions accept FormData because forms submit FormData. Parse images JSON from a hidden field. Price must be converted: the form shows NOK (e.g., "299") but storage is in ore (29900). Multiply user input by 100.

**src/actions/experiences.ts** (ADMN-03):
Similar pattern. Functions:
- `createExperience(formData: FormData)` — Write to `experiences` collection. Also create subcollection docs in `experiences/{id}/dates/` for each date entry. Each date doc: `{ date: Timestamp, maxSeats, bookedSeats: 0, availableSeats: maxSeats, isActive: true, priceOverride: null }`.
- `updateExperience(id: string, formData: FormData)` — Update main doc. For dates: compare with existing dates — add new ones, update changed ones, delete removed ones. Use batch write for atomicity.
- `deleteExperience(id: string)` — Delete all dates subcollection docs first, then delete main doc. Call `revalidateTag('experiences')`.
- `getAllExperiences()` — No publishedAt filter for admin list.

**src/actions/articles.ts** (ADMN-04):
Functions:
