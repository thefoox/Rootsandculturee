# Phase 3 — Betaling og Booking: UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This spec extends Phase 1 and Phase 2 UI-SPECs. All tokens, breakpoints, interaction states and accessibility rules from those phases carry forward unchanged. Only Phase 3 additions and overrides are declared here.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — Tailwind v4 CSS custom properties (continues Phase 1/2 pattern) |
| Preset | not applicable |
| Component library | none — hand-built components on semantic HTML + Tailwind v4 |
| Icon library | lucide-react (tree-shakeable, accessible) |
| Font — Headings | Merriweather (serif), loaded via `next/font/google` |
| Font — Body | Inter (sans-serif), loaded via `next/font/google` |
| New dependency | `@stripe/stripe-js` + `@stripe/react-stripe-js` — Stripe Elements inline payment form |
| New dependency | `resend` — transactional email (server-only, no UI surface) |

**Phase 3 exceptions:**

## Surface Area 1: Shopping Cart

### Cart Icon (Header — update from Phase 2 placeholder)

The existing `<Header>` `ShoppingBag` icon placeholder is upgraded with a live badge.

**Badge:**
- Absolute positioned top-right on the icon button container
- Size: 18px diameter circle (`w-4.5 h-4.5` via inline style or Tailwind arbitrary `w-[18px] h-[18px]`)
- Background: `#C0392B` (cart-badge token)
- Text: `#FEFCF3`, Inter 13px 400, centered, `font-variant-numeric: tabular-nums`
- Hidden when cart is empty (no badge renders, no empty "0")
- `aria-label` on cart link updated to: `"Handlekurv, {N} varer"` when count > 0, `"Handlekurv, tom"` when count = 0
- `aria-live="polite"` region wrapping badge so screen readers announce count changes

### Cart Drawer (side panel)

**Trigger:** Clicking the cart icon in `<Header>` opens the drawer (instead of navigating to `/handlekurv`). Navigating directly to `/handlekurv` also available as full page.

**Drawer container:**
- `position: fixed; right: 0; top: 0; bottom: 0; z-index: 150` (above header z-50)
- Width: 400px on desktop, 100vw on mobile (≤ 400px viewport)
- Background: `#FEFCF3`
- `border-left: 1px solid rgba(27, 67, 50, 0.12)`
- `box-shadow: -4px 0 24px rgba(0,0,0,0.10)`
- Slide-in animation from right: `transform: translateX(100%)` → `transform: translateX(0)`, 200ms ease-out, `motion-safe:` wrapped
- Backdrop: `rgba(27,67,50,0.4)` full viewport overlay behind drawer, closes drawer on click

**Drawer layout:**
```
[Drawer header — 64px, border-b border-forest/12]
  "Handlekurv" — Merriweather 20px 700, forest, left-aligned
  Close button — X icon (24px lucide), forest, 44px touch target, aria-label="Lukk handlekurv"

[Cart item list — flex-1, overflow-y: auto, p-16 gap]
  [CartItem component × N]
  [Empty state — centered, when cart is empty]

[Cart footer — sticky bottom, border-t border-forest/12, p-16]
  [Order summary: subtotal, shipping estimate]
  ["Gå til betaling" button — primary, full width]
  ["Se full handlekurv" — ghost, full width, links to /handlekurv]
```

**ARIA:**
- `role="dialog"`, `aria-modal="true"`, `aria-label="Handlekurv"`
- Focus trap: focus moves to close button on open; returns to cart icon trigger on close
- Escape key closes drawer

**Layout:** Flex row, 80px minimum height, `py-12`, `border-b border-forest/08`, `gap-12`

**Product thumbnail:** 64px × 64px, `object-fit: cover`, `border-radius: 4px`

**Item content (flex-1):**

**Remove button:**

**Item total price:** Inter 15px 400, `#A0522D` (rust), right-aligned, bottom of item row

**Route:** `/checkout`

**Layout:**
- Standard public layout (Header + Footer, max-width 900px — narrower than catalog for focus)
- Two-column on desktop: checkout form left (60% width), order summary right (40% width), 48px gap
- Single column stacked on mobile: form top, order summary below (order summary collapses to accordion on mobile)

**Page heading:** `<h1>` Merriweather 28px 700, forest, `pt-48 pb-32`

**Checkout form sections:**

Section 1 — "Kontaktinformasjon":
- E-postadresse: `<Input>` component (Phase 1 pattern)
- For logged-in users: e-post pre-filled, field read-only with "Innlogget som {email}" label

Section 2 — "Leveringsadresse" (for product items; hidden if cart contains only experience bookings):
- Fullt navn: `<Input>`
- Adresse: `<Input>`
- Postnummer: `<Input>`, type="text", pattern="[0-9]{4}" (Norwegian 4-digit postal code)
- Sted: `<Input>`
- All fields have standard `<Input>` label-above + error-below pattern (Phase 1)

Section 3 — "Betaling":
- Section heading: Merriweather 20px 700, forest
- Stripe lock icon (LockKeyhole lucide, 16px, bark) + "Sikret med Stripe" — Inter 13px 400, bark, flex row, `mb-16`
- Stripe Elements container: `bg-card`, `border border-forest/20`, `border-radius: 8px`, `p-24`
- Stripe `<PaymentElement>` renders inside — Stripe's own UI. Theme configuration (see Stripe Elements Styling below)
- Below Stripe Elements: `<FormError>` for payment errors (Stripe decline codes, network errors)

**Sticky order summary (right column / mobile accordion):**
- Same panel design as cart page order summary
- Shows itemized list: product names + quantities OR booking name + date
- Shows subtotal, shipping, total
- Not editable from checkout (user must go back to cart to modify)

**Submit button:**
- "Betal nå" — `Button variant="primary"`, full width, LockKeyhole icon + label
- Loading state: "Behandler betaling..." with spinner, `aria-busy="true"`, `disabled`
- Only enabled when Stripe Elements form is complete (Stripe's `complete` state)

**Guest vs. logged-in:** No separate guest/logged-in checkout routes. A single `/checkout` page detects session state. Logged-in users see pre-filled email + a "Koble til konto" note; guests proceed with email entry only.

### Stripe Elements Styling

Stripe Elements accepts a `theme` object in `loadStripe` appearance options. Configure to match brand:

```javascript
// Stripe Elements appearance — matches Roots & Culture palette
const appearance = {
  theme: 'flat',
  variables: {
    colorPrimary:       '#B84D00',  // ember — accent
    colorBackground:    '#F5F0E8',  // card — matches container background
    colorText:          '#1B4332',  // forest — body text
    colorDanger:        '#C0392B',  // destructive — error messages
    fontFamily:         'Inter, system-ui, sans-serif',
    fontSizeBase:       '15px',
    borderRadius:       '4px',
    spacingUnit:        '4px',
  },
  rules: {
    '.Input': {
      border: '1px solid rgba(27, 67, 50, 0.2)',
      boxShadow: 'none',
    },
    '.Input:focus': {
      border: '1px solid #1B4332',
      boxShadow: 'none',
      outline: '2px solid #1B4332',
      outlineOffset: '2px',
    },
    '.Label': {
      color:       '#1B4332',
      fontSize:    '13px',
      fontWeight:  '400',
    },
  },
}
```

**Note:** Stripe Elements renders in a sandboxed iframe. The appearance object is the only styling interface. Do not attempt to CSS-target Stripe iframe internals.

## Accessibility Contract (Phase 3 extensions)

All Phase 1 and Phase 2 WCAG requirements carry forward. Phase 3 additions:

| Requirement | Implementation |
|-------------|---------------|
| Cart drawer focus trap | Focus enters close button on open; Tab cycles within drawer; Shift-Tab reverse; Escape closes; focus returns to cart icon trigger on close |
| Cart badge live region | `<span aria-live="polite" aria-atomic="true">` wrapping badge count; screen reader announces "Handlekurv, 3 varer" on update |
| Date card group | `role="group"` with `aria-labelledby` on the date picker section; each card is `<button type="button">` with `aria-pressed` |
| Sold-out date card | `disabled` attribute; `aria-disabled="true"`; "Utsolgt" text visible + redundant `aria-label="[date], utsolgt"` on button |
| Capacity block | `role="alert"` on the "alle plasser er fylt" block — announces to screen readers when capacity is reached |
| Stripe Elements | Stripe's `<PaymentElement>` is WCAG-compliant internally; ensure surrounding container has `aria-label="Betalingsinformasjon"` as region |
| Checkout form | All `<Input>` fields use Phase 1 `aria-describedby` pattern; Zod errors propagate to `<FormError>` components |
| Confirmation modal | `role="dialog"`, `aria-modal="true"`, `aria-labelledby`; focus trapped; no accidental dismiss; confirmation code in `<code>` with descriptive `aria-label` |
| Confirmation code | `<code aria-label="Bekreftelseskode: {code}">` or wrapped in a `<p>` with adjacent visible label |
| Admin order status update | Toast announcement on save: `role="status"`, `aria-live="polite"` |
| Admin booking cancellation | DeleteConfirmDialog reused — inherits Phase 2 focus trap + ARIA contract |
| Admin filter selects | Each `<select>` has a `<label>` with `htmlFor` association; not styled as custom dropdowns |
| Cart item removal | Removal is reversible (user can re-add); no confirmation dialog needed; toast confirms: "Fjernet fra handlekurven." (`role="status"`) |
| "Hva du ma ta med" checklist | `<ul>` with `<li>` items; CheckSquare icon is `aria-hidden="true"`; text is the actual list content |

- **Betal nå** — no confirmation dialog. Stripe Elements handles payment intent confirmation. Loading state + error recovery handles failure cases.
- **Oppdater ordrestatus** — no confirmation dialog. Action is reversible (admin can change status again). Toast confirms success.
