    **2. Create server actions for profile mutations:**

    **src/actions/profile.ts** (NEW) — Two server actions:

    `updateProfileAction(formData: FormData)`: Verify session with `verifySession()`. Extract `displayName` and `address` from formData. Validate with Zod: displayName required (min 2 chars), address optional string. Update Firestore `users/{uid}` doc with `adminDb.collection('users').doc(uid).update(...)`. Return `{ success: true }` or `{ success: false, error: string }`. All error messages in Norwegian. Per D-09: e-post is read-only (locked by Firebase Auth) -- do NOT allow email changes.

    `changePasswordAction(formData: FormData)`: This action receives `currentPassword`, `newPassword`, `confirmPassword`. Validate with Zod: newPassword min 8 chars, confirmPassword must match newPassword. Return `{ success: true, requiresReauth: true }` to signal the client to perform the actual password change client-side using Firebase Auth `reauthenticateWithCredential` + `updatePassword` (per D-11: requires old password verification). The server action only validates -- actual password change must happen client-side since Firebase Auth `updatePassword` requires the current user's auth state. Return error messages in Norwegian.

    NOTE on password change architecture: Firebase Auth password change MUST happen client-side. The server action validates form data and returns a signal. The client component (Task 2) handles the actual Firebase Auth calls.
